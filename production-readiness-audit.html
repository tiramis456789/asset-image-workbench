프로덕션 준비도 코드 감사

브라우저와 데스크톱 셸을 함께 쓰는 이미지 정리 도구를 대상으로, 저장소 안의 소스·테스트·문서·설정·빌드 산출물 흔적까지 전수 검토해 실제 배포 관점의 리스크를 평가했다. 판단 기준은 도구 보유 여부가 아니라, 이 프로젝트의 가장 취약한 표면이 실제로 보호되고 있는지다.

A. 개요 및 스택

이 코드는 로컬 이미지 폴더를 읽어 가상 이름 변경, 가상 폴더 이동, EXIF 기반 분류를 수행한 뒤 마지막 단계에서 실제 파일 시스템에 반영하는 데스크톱 우선 워크벤치다. 호스팅형 서비스가 아니라 개인용 로컬 도구에 가깝지만, 실제 파일 쓰기와 Electron 패키징이 포함되어 있어 “로컬이라서 괜찮다”로 넘길 수 없는 안전 요구가 있다.

감지된 기술

TypeScript 5.8 React 18 Vite 5 Vitest 3 Electron 37 Tailwind CSS 3 Radix UI / shadcn 계열 래퍼 File System Access API

검증된 정량 지표

  • 소스 파일: 97개. 산출 기준은 src와 electron 아래의 .ts, .tsx, .js, .mjs, .css 파일이다.
  • 총 소스 라인: 9,740줄. 이 중 앱/라이브러리 코드는 8,851줄, 테스트는 797줄이다.
  • 테스트 파일: 8개, 테스트 케이스: 29개. 근거는 src/test와 실제 vitest run 결과다.
  • 대형 파일: 300줄 초과 파일이 최소 8개이며, 핵심 상태 저장 훅 src/hooks/useImageStore.ts는 524줄, EXIF 파서는 src/lib/exif.ts 450줄, 우측 작업대는 src/components/viewer/RenameWorkbench.tsx 402줄이다.
  • 커밋 수: 산정 불가. 현재 작업본에는 .git 디렉터리가 없어 커밋 이력과 브랜치 이력을 검증할 수 없었다.

D. 실행 계획

  1. 하드닝 — electron/main.mjs에서 no-sandbox, disable-gpu-sandbox, sandbox: false를 제거하고 필요한 브리지 권한만 preload로 노출하라.
  2. 하드닝 — src/hooks/useImageApply.ts의 파일 반영을 임시 파일 쓰기 → 검증 → 원본 교체 → 실패 시 롤백 가능 흐름으로 바꿔 실제 폴더 손상 위험을 줄여라.
  3. 빠른 개선 — tsconfig를 단계적으로 강화해 최소한 strictNullChecks와 noImplicitAny부터 켜고, 새 코드에 대한 타입 예외를 줄여라.
  4. 빠른 개선 — package.json과 package-lock.json의 이름·버전을 맞추고, npm 또는 bun 중 하나만 공식 경로로 남겨라.
  5. 빠른 개선 — src/index.css의 외부 폰트 @import를 제거하고 폰트를 번들에 포함하거나 시스템 폰트로 대체하라.
  6. 전략 과제 — useImageStore를 파일 적용 도메인, 트리 조작 도메인, 뷰어 선택 상태로 분해해 기능 추가 시 중심 훅의 폭발 반경을 줄여라.
  7. 전략 과제 — 로컬 통과에만 의존하지 말고 CI에 lint, test, build를 묶어 최소 배포 게이트를 만들어라.

핵심 사용자 문제는 정확히 붙잡았고 훅 단위 검증도 해놨지만, 프로토타입 속도를 우선한 흔적이 구조·환경·Electron 보안 하드닝보다 앞서 있는 코드베이스다.

F. 최종 판정

이 프로젝트는 로컬 이미지 묶음을 검토하고 이름과 위치를 가상으로 조정한 뒤 마지막에 실제 폴더에 반영하는 데스크톱형 작업 도구다. 서버 제품은 아니지만 사용자 파일을 직접 만지는 만큼, 안정성과 데스크톱 보안 태세를 가볍게 볼 수 없는 범주에 속한다.

솔직히 말하면, 이 코드는 “아이디어 검증을 넘긴 괜찮은 개인용 툴”이지 “지금 바로 배포해도 마음 편한 제품”은 아니다. 좋은 점은 분명하다. 핵심 파일 작업과 이름 변경 흐름에 대한 테스트가 실제로 있고, 빌드와 린트도 현재는 통과하며, 사용자에게 변경을 마지막 단계까지 가상으로 유지하는 제품 판단도 좋다. 하지만 Electron 샌드박스를 꺼버린 순간 보안 점수는 급락하고, 파일 반영이 원자적이지 않은 순간 운영 신뢰도도 제한된다. 여기에 느슨한 타입 설정, 락파일 불일치, 과한 생성 UI 래퍼까지 겹치면 팀 개발이나 외부 배포로 확장할 때 발목을 잡는다.

프로덕션 준비 판정: 지금 상태로는 배포 보류가 맞고, Electron 보안 하드닝과 파일 적용 경로 재설계가 선행되어야 한다.
